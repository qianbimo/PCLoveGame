## 2. 背景与目标

**背景**：项目计划开发一款运行于 Windows PC 端的像素风互动闯关游戏，使用 Unity 2D 作为主要开发引擎，通过二维像素场景、角色动画、剧情分支、小游戏机制和好感度变化，构建具有完整情感线的恋爱剧情互动体验。整体剧情围绕一名男主与一名女主展开，以“五个互动关卡”推进故事，玩法覆盖 21 点对战、校园邀约、观影答题、恋爱破碎剧情与拼图和好结局。

**目标**：实现一款完整可运行的 PC 端 2D 像素风恋爱剧情互动游戏，具备主菜单、关卡流程、剧情对话、角色动画、五个关卡玩法、好感度系统、关卡解锁机制以及本地存档功能，并支持窗口化与全屏模式运行。

## 3. 范围

- **本期范围**：
  - 实现主菜单、继续游戏、关卡选择、设置、退出等基础入口
  - 实现五个关卡的剧情与玩法流程
  - 实现统一的像素风 UI、角色动画、场景切换与反馈
  - 实现好感度系统、关卡解锁与进度保存
  - 实现本地存档、基础设置项持久化
  - 支持 Windows PC 端运行

- **非本期范围**：
  - 联网功能、排行榜、账号系统
  - 云存档与跨设备同步
  - 敏感个人信息采集与用户数据平台
  - 多平台发布（移动端、主机端、网页端）
  - 明确的多结局复杂分支体系
  - CG 图鉴、成就系统、角色换装、剧情回放等扩展内容

## 5. 非功能需求

- **性能需求**：游戏应在普通 PC 上流畅运行，目标帧率为 60 FPS，UI 点击即时响应，拼图拖拽过程中无明显卡顿。
- **兼容性需求**：支持 Windows 10 及以上系统，兼容 `1920x1080`、`1366x768` 等常见分辨率，并支持窗口化与全屏模式。
- **易用性需求**：操作方式应简单直观，主要使用鼠标点击、键盘输入与拖拽交互；各关卡开始前应提供玩法说明；错误输入应给出明确反馈。
- **可维护性需求**：代码需按模块划分；剧情文本、关卡配置、选项数据与拼图配置建议配置化管理，避免逻辑与数据高度耦合。
- **可扩展性需求**：后续应可扩展更多关卡、分支结局、角色换装、CG 图鉴、成就系统和剧情回放功能。
- **安全性需求**：本地存档仅保存游戏进度和设置，不应保存敏感个人信息。
- **风格一致性需求**：角色、场景、UI、字体、按钮、弹窗、转场动画和反馈特效应保持统一像素风格，避免风格割裂。

## 6. 数据、接口与约束

- **核心数据**：
  - 关卡数据：`levelId`、`levelName`、`isUnlocked`、`isCompleted`、`nextLevelId`
  - 对话数据：`speaker`、`content`、`animationKey`、`nextDialogueId`
  - 选项数据：`choiceId`、`choiceText`、`resultText`、`favorChange`、`targetScene`
  - 拼图数据：`pieceId`、`startPosition`、`targetPosition`、`isPlaced`、`snapDistance`
  - 存档数据：`currentLevel`、`unlockedLevel`、`favorability`、`isGameCleared`

- **外部接口与依赖**：
  - 运行时引擎依赖 Unity 2D 和 C#
  - UI 依赖 Canvas、Button、Image、TextMeshPro、TMP_InputField
  - 角色动画依赖 Sprite Renderer 与 Animator Controller
  - 拼图拖拽依赖 Unity EventSystem，如 `IBeginDragHandler`、`IDragHandler`、`IEndDragHandler`
  - 本地持久化依赖 JSON 文件与 `PlayerPrefs`

- **业务规则与约束**：
  - Game 1 为三局制 21 点对战，超过 21 点视为爆牌失败
  - Game 2 中仅选项 C “一起去上网”能推进剧情
  - Game 3 中电影名称与时间均正确才可通关
  - Game 4 不存在正确选项，三项均导致好感度下降
  - Game 5 中错误放置拼图不扣好感度，正确放置逐步恢复好感度，完成后恢复至 100/100
  - 玩家只能通过关卡选择进入已解锁关卡
  - 对话、选项、关卡配置建议使用 ScriptableObject 或 JSON 配置，避免全部写死在代码中

## 7. 风险与待确认

- **风险与依赖**：
  - 像素风角色立绘、序列帧动画、背景与 UI 资源制作成本较高，可能成为开发进度瓶颈
  - 五个关卡玩法差异较大，若缺少统一的状态管理和数据结构，后期维护成本会明显上升
  - 剧情表现依赖动画、台词、音频和转场的联动，编排不当可能影响整体体验
  - 拼图拖拽、吸附阈值和输入判定等交互细节若调优不足，容易影响用户手感

- **待确认事项**：
  - Game 3 的电影名称标准答案“如果声音不记得”是否为最终正式文案，是否需要支持别名或模糊匹配
  - Game 4 中好感度在进入第五关前的初始值与扣减逻辑，是否固定为文档所述流程，还是需根据前置表现动态计算
  - Game 5 拼图碎片数量、布局复杂度、吸附阈值及进度增长规则尚未细化
  - 结局场景是否只有单一“和好结局”，还是后续会扩展失败结局、隐藏结局或分支结局
  - 存档结构是否需要记录玩家选项历史、剧情节点和已播放演出状态
